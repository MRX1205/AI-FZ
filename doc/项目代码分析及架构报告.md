## 1. 项目概述

高翠网是一个 **翡翠找货平台**。它将 AIGC 能力深度融入传统的电商或客资收集模式中。
项目整体包含面对买家的 **AI 导购前台** 以及面向商家的 **后台管理端**。

*   **对买家**：弱化传统的商品列表浏览结构，引入核心大模型对话机制，通过 AI 助手（Jade Agent）提取用户的翡翠需求（预算、品类、尺寸等），从而实现更精准的找货。
*   **对商家**：提供支持商品发布、会员体系（VIP等级与客资查看权限限制）以及 AI 商品草案生成的管理控制台。

---

## 2. 整体系统架构与技术栈

项目采用了典型的 **前后端分离架构（B/S）** 设计：

### 2.1 后端技术栈 (FZ-backend)
*   **核心框架**：Python 3.12+ 配合 **FastAPI**，提供高性能的异步 RESTful 接口。
*   **ORM 与数据库**：使用 **SQLAlchemy 2.x** 配合 **PostgreSQL**。结合 **Alembic** 用于数据库迁移。
*   **数据向量化**：采用了 **pgvector** 插件，结合了阿里云的 **DashScope (通义千问 Embeddings)** 将商品信息向量化并做近似匹配搜索。
*   **AI 接口**：项目中广泛集成了外部大语言模型。主要由 `httpx` 发起异步 HTTP 调用，默认对接模型服务名为 MiMo / DashScope。
*   **第三方能力对接**：
    *   **Supabase OTP**：邮箱无密码登录发送及校验邮件验证码。
    *   **支付宝 (Alipay)**：对接支付体系，控制商家的免费 / VIP 支付开通流程。
    *   **MinIO / 本地静态池**：用户及商品图片等资源静态服挂载。

### 2.2 前端技术栈 (FZ-front)
*   **核心框架**：**React 19 + TypeScript + Vite**，主要面向移动端（H5 比例布局，带外壳模拟限制宽度在 430px 内）设计。
*   **状态与路由**：**React Router v7** 进行单页面路由管理，未使用 Redux/Zustand 等复杂业务级全局状态库，采用局部 state + Context 以及 `localStorage`。
*   **API 客户端**：原生 `fetch` 结合 TS 泛型做基础封装通信。
*   **样式方案**：纯原生或模块化 CSS 机制，未使用比如 Tailwind CSS 之类的预处理包。UI Icon 主要采用包 `lucide-react`。

## 3. 后端代码文件结构与模块分析 (FZ-backend)

进入 `FZ-backend` 文件夹下，包含 `app`, `alembic` 迁移文件夹等。

### 3.1 核心网络与应用入口库
*   `app/main.py`: 后端 FastAPI 应用实例化入口，挂载跨域中间件(CORS)、将`/uploads`转为本地静态文件资源池，并注册了路由 API 挂载点`/api`。
*   `app/core/config.py`: 依赖 `pydantic-settings` 统一读取环境变量 `.env`，管理如支付宝网关、MiMo API Key、DashScope 密钥、JWT 秘钥等敏感信息。

### 3.2 数据库及模型映射 (app/models & app/db)
*   `app/db/session.py` / `base.py`: 建立异步 SQLAlchemy 引擎、维持长连接池并暴露 Base。
*   `app/models/*.py`: 
    *   `auth.py` / `merchant.py`: 管理商家的基本鉴权体系及属性字段。
    *   `chat.py`: 买家（Visitor）的会话组和消息持久化模型。
    *   `lead.py`: 管理买家的询价、找货意图（客资线索）。
    *   `payment.py`: VIP 的充值订单持久化。
    *   `product.py`: 记录商家的商品详情(`MerchantProduct`)、图片(`MerchantProductImage`)，以及**核心特征**：`MerchantProductEmbedding`，将大段产品语料进行`[1024]`维向量编码存储 `Vector(1024)`。

### 3.3 业务服务层服务应用 (app/services)
该目录下承担了平台几乎所有的厚重逻辑：
*   **`jade_agent.py` (AI找货助手)**: 定义了“高翠AI”核心，封装大模型对话流，内置了限制提示词 `MIMO_SYSTEM_PROMPT` 去追问预算、品相而不直塞虚假商品。
*   **`product_image_recognition_agent.py` (识图生成商品)**: 供商家上传商品图时流转至 AI 系统进行理解归纳的 Agent。读取本地的 `UPLOAD_BASE` 图转码传给大模型并产出“匹配参数字典”与草稿文案。
*   **`embeddings.py`**: 将 `product.py` 中的商品详情串，调用阿里云 Dashscope 的 Embedding API 提取浮点化向量 `EmbeddingResult`。
*   **`product_search.py` & `visitor_product_matcher.py`**: 利用上述生成的向量，在 PG 库中使用 HNSW 或欧氏距离算法来对买家的需货要求在几十万件商品内做近临精准推送。
*   **`supabase_otp.py`**: 集成防流控的无密码邮箱注册或回环登录逻辑。
*   **`alipay.py` & `vip_orders.py`**: 用于生成签名发送扫码付链接并异步接听蚂蚁金服服务器 `notify_url` 回调回调更改账户 `VIP` 权限。

### 3.4 API 控制器路由层 (app/api/routes)
承上启下的 HTTP 接口端：
*   `auth.py`: 提供给前端邮箱验证码发送校验，换发内部系统 JWT 令牌。
*   `chat.py`: 处理通过 visitorId 发起的 `/chat` 相关操作并串联 `jade_agent.py` 返回回答流。
*   `merchant.py`: 商家资料或客源(lead)获取。拦截免费用户，展示邮箱星号掩码等逻辑限制。
*   `products.py`: 控制商品草稿、上下架及匹配修改流程。
*   `payments.py`: 发起 VIP 付款，接收 callback 解除商家客源屏蔽。

## 4. 前端代码文件结构与模块分析 (FZ-front)

移动优先、高度拆解的单页应用：

### 4.1 全栈启动和核心环境定义
*   `src/main.tsx` & `src/App.tsx`: 初始化系统并使用 React-Router 的 `Routes`。根据是否开发已完毕用 `PlaceholderPage` 对未来页降级托底展示。
*   `src/routes.ts`: 硬编码存储了两大分支“买家端”和“商家端”的可达路由，用于整体系统结构映射。

### 4.2 API 客户端 (src/api)
*   `src/api/client.ts`: 对后端接口进行强类型包装，例如统一带上 Authorization Bearer，并且约定了 401 自动触发重定向返回登录等常规前置阻断。

### 4.3 核心功能页面组件 (src/pages)

前端切分成两大块： **【买家寻货生态】** 与 **【商家管理流】**

*   **HomePage / ProductDetailPage** *(买家侧)*：
    *   **HomePage**: `VisitorId` 生成存储地，负责渲染主会话。将买家对翡翠的参数投喂给后端接口获取回答。
    *   **ProductDetailPage**: 展示系统匹配的推荐商品详情。
*   **MerchantAuthPage** *(商户鉴权)*：
    *   纯依靠接收 `Supabase` 发到的邮件触发验证码登录流程，登陆后写入 token。
*   **MerchantDashboardPage** *(商户仪表盘)*：
    *   高屏占比，首页面显示免费/VIP标识、今日可转化客资源及存量商品数量，实现“重流量展示”。
*   **MerchantLeadsPage / MerchantLeadDetailPage** *(客源处理)*：
    *   商家捕获到的需求工单（Lead），受后端数据拦截后，非 VIP 无法获取真实邮箱。
*   **MerchantPublishPage / MerchantPublishEditPage** *(智能上云)*：
    *   **极具特色的发布流程**：上传图片(MerchantPublishPage) -> `product_image_recognition_agent` 分析并生成带有 `match_params` 结果页(ResultPage) -> 人工校验补全后执行入库上架 (EditPage)。
*   **MerchantAccountPage...** 等其余页围绕充值状态回调、个人配置等基础建设。

## 5. 关键业务工作流程解析

### 工作流 1：买家智能找货闭环
1. 买家打开前端由于未携带任意标示，被认为是访客 `Visitor`。
2. 前端随机生成一套设备追踪 Id 发射给 `/api/chat` 完成会话落点。
3. 聊天界面询问，例如：“找一个五万左右的冰种阳绿手镯”。后端在 `chat.py` 中记录，随即将消息链传递至大模型（`jade_agent.py`）。
4. 在需要推货阶段（触发特定逻辑或离线计划），抽取买家画像送入 `visitor_product_matcher.py`，再结合 PGVector 拿需求文本和翡翠商品的 `[1024]维度向量` 比对获取余弦相似或 L2 逼近的单品返回给界面进行点击。

### 工作流 2：商家商品 AI 极速上架闭环
1. 商家在移动 H5 进入 【发布商品】模块，连拍翡翠原石或加工戒面照片上传至后端。
2. 后端保存文件至 `uploads` 后，调用系统封装的 `Jade Agent (识图子类)` 发送 `PRODUCT_IMAGE_RECOGNITION_PROMPT`；AI 直接剥离出预算、尺寸乃至可能的翡翠种水信息。
3. 将识别后的非确定性要素填充进 `draft` 草稿并返回。前端跳往【发布结果】页面。
4. 商家复核数据更正“未知”项并点发布。
5. 后端最终将结果序列化生成 Vector 编码推到 PostgreSQL 落库。从开始到完成上架一气呵成。

### 工作流 3：分级商户及支付解锁信息闭环
1. 系统存在免费用户；后台能够正常拦截客资意向(`Leads`)，但核心字段如买家 Email 会被遮罩并在界面透出升级暗示。
2. 商户决定开通权限，在 `AccountBenefits` 发起开通 VIP 按钮请求。
3. 后端 `payments.py` 中经由 `alipay.py` `[AliPayClient]` 创建 Alipay Oauth 并签发的 Trade url 让用户端跳转应用外跳。
4. 用户扫脸付清离开 Alipay 环境。支付宝服务器向后端发送 `Server notify` 回调。后端使用 `cryptography` 包对其 RSA 公钥解密验签确定充值通过。
5. 后端调用 `merchant_membership.py` 修改商户 VIP 时间与等级，商户刷新 Leads 页，数据锁屏蔽自动解除。

## 6. 总结
**高翠网本质上搭建了一套依靠“NLP 生成式闲聊”和“向量空间近邻搜索”代替传统“树形品类与搜算引擎”的买家找货平台。**
从代码可以看出后端在数据交互、大模型上下文保护做了大量的防御性工程设计；且引入 PostgreSQL/PgVector 将常规型业务做到了前沿向量召回支撑。系统的脚手架结构合理，扩展商品模型或接入如文生图、短视频智能解说等更多 AI Agent 将大有可为。
