## 2. 项目一句话概括

这是一个面向翡翠货源撮合的移动端 H5 系统：

- 买家端：通过 AI 聊天描述需求，系统解析需求并匹配商家商品；
- 商家端：通过邮箱验证码登录，完成商品发布、商品管理、客资查看、通知查看、VIP 升级和个人中心管理；
- 后端：用 FastAPI 承担认证、聊天、商品、客资、支付、文件上传、AI 调用、向量匹配；
- 数据层：PostgreSQL + pgvector 保存业务数据和商品 embedding；
- AI 能力：MiMo 用于聊天和图片识别文案生成，DashScope 用于文本 embedding。

## 6. 环境配置与部署关系

### 6.1 本地开发配置

后端代码实际读取的是：

- `FZ-backend/.env.bak.codex`

不是常见的 `FZ-backend/.env`。这意味着：

- README 里部分说明和常规直觉不完全一致；
- 如果开发者只改 `.env`，后端可能不会读到；
- 这是一个很容易踩坑的环境约定。

本地数据库按当前后端配置应走：

- `127.0.0.1:55432`

这与 `docker-compose.local.yml` 是一致的。

### 6.2 仓库里的生产部署配置

仓库部署文件默认设计为：

- 前端容器暴露 `6140`
- 后端容器暴露 `6141`
- Nginx 把 `/` 代理到 `6140`
- Nginx 把 `/api/` 和 `/uploads/` 代理到 `6141`

见：

- `deploy/docker-compose.prod.yml`
- `deploy/nginx.merchant.lyhlz.cn.conf`

### 6.3 你提供的线上现状

你明确补充了当前线上现状：

- 前端域名：`https://merchant.lyhlz.cn`
- 后端端口：`6145`
- 数据库：已经在服务器中运行，端口和本地一致

这说明：

- 线上真实运行方式和仓库里现有部署文件不是同一套；
- 仓库默认写的是 `6141`，你线上实际是 `6145`；
- 后续任何运维排查，都必须优先以线上实际配置为准，而不是默认拿仓库里的 `deploy` 当现状。

### 6.4 配置漂移结论

当前项目存在“代码逻辑基本统一，但部署配置有多套”的情况：

- 本地开发：`55432`
- 仓库生产 compose：`6140/6141`
- 你口头确认线上：后端 `6145`

后续最好统一一份“线上事实配置说明”，否则排查 bug 时很容易误判。

## 8. 数据模型设计

### 8.1 商家与认证

#### `merchants`

作用：

- 保存商家主体信息
- 保存会员等级
- 保存 VIP 起止时间
- 保存通知偏好

关键字段：

- `email`
- `tier`
- `vip_started_at`
- `vip_expires_at`
- `web_notification_enabled`
- `email_notification_enabled`

#### `merchant_sessions`

作用：

- 保存商家登录会话 token

实现方式：

- 登录成功后生成 UUID token
- 前端把 token 存 localStorage
- 后端通过 `Authorization: Bearer xxx` 查 `merchant_sessions`

#### `auth_codes`

当前代码里保留了这个表，但实际登录验证码流程已经外包给 Supabase OTP。

也就是说：

- 表结构还在
- 当前主要认证路径不再依赖这张表

这是典型的“历史设计保留，运行主路径已切走”的状态。

### 8.2 商品相关

#### `merchant_products`

作用：

- 商家商品主表

保存内容：

- 标题
- 简介
- 详情
- 标签
- 价格
- 状态
- 图片 URL 列表
- 匹配参数
- 搜索文本
- 发布时间

状态有三种：

- `draft`
- `listed`
- `unlisted`

#### `merchant_product_images`

作用：

- 保存每张商品图的真实上传记录

和 `merchant_products.image_urls` 的关系：

- `image_urls` 更像页面展示快照
- `merchant_product_images` 是结构化图片表

项目当前两套同时存在。

这让系统兼容了：

- 旧的 mock 图 URL
- 新的上传图片记录

但也意味着要特别注意两套数据一致性。

#### `merchant_product_embeddings`

作用：

- 保存商品 embedding

字段包括：

- provider
- model
- dimensions
- content_hash
- embedding

它支撑买家端向量匹配。

### 8.3 客资与通知

#### `merchant_leads`

作用：

- 保存买家留资

字段包括：

- 哪个商家

注意这里保存了商品快照，而不是完全依赖商品主表，目的很明显：

- 即使商品后续改名或下架，客资记录仍能保留当时上下文。

#### `merchant_notifications`

作用：

- 保存商家系统通知

目前通知类型主要有：

- `new_lead`

### 8.4 聊天与需求画像

#### `chat_messages`

## 9. 后端服务层拆解

### 9.1 `supabase_otp.py`

作用：

- 调 Supabase OTP 发送邮箱验证码
- 校验邮箱验证码

工作方式：

1. 拼出 `/auth/v1/otp`
2. 发邮箱验证码
3. 登录/改邮箱时调 `/auth/v1/verify`

特点：

- 业务邮箱验证码不自己发
- 不自己生成验证码
- 不自己维护验证码生命周期

### 9.2 `merchant_membership.py`

作用：

- 计算商家是否“有效 VIP”

逻辑：

- 即使 `tier == vip`
- 只要 `vip_expires_at` 过期
- 系统也按 `free` 处理

这个服务被很多页面和接口复用：

- 商品额度
- 客资是否可见全邮箱
- 通知类型
- 排序权重

### 9.3 `jade_agent.py`

作用：

- 封装 MiMo 文本聊天调用

用途：

- 买家普通聊天
- 图片识别生成商品文案
- 游客需求改写
- 游客可见回复生成

特点：

- 统一走 `chat/completions`
- 显式关闭 thinking
- 出错时返回友好兜底文案

### 9.4 `product_image_recognition_agent.py`

作用：

- 根据商家上传的商品图片生成商品草稿

生成字段：

- `title`
- `summary`
- `detail`
- `tags`
- `priceCents`
- `matchParams`

工作流程：

1. 前端上传图片
2. 后端把图片保存到 `/uploads`
3. 服务把图片转成 base64 data URL
4. 调 MiMo 多模态能力
5. 要求返回 JSON
6. 解析 JSON 并截断长度
7. 清洗价格和标签
8. 写回商品草稿
9. 同步刷新 `search_text`

它不是“直接发布”，而是“先生成草稿供商家确认”。

### 9.5 `product_search.py`

作用：

- 统一生成商品搜索文本
- 统一标准化 `match_params`
- 为 embedding 提供输入内容

也就是说，embedding 不是直接拿标题做，而是拿一段拼装后的、较完整的商品语义描述做。

### 9.6 `embeddings.py`

作用：

- 调 DashScope 生成 embedding

使用场景：

- 商品发布或编辑后，若商品已上架，需要刷新 embedding
- 买家需求匹配时，需要给需求文本生成 embedding

### 9.7 `visitor_product_matcher.py`

这是整个买家端最关键的服务。

它做了四件事：

1. 判断用户是不是直接在找翡翠
2. 如果不是，先把原话改写成翡翠购买需求
3. 把需求解析成结构化画像
4. 用向量 + 规则找候选商品，再生成用户可见回复

#### 9.7.1 需求解析

系统会从自然语言中抽：

- 预算
- 品类
- 颜色
- 种水
- 器型
- 瑕疵
- 用途
- 证书要求
- 关键词

例如：

- “预算 2 万”
- “冰种”
- “平安扣”
- “无纹无裂”
- “送礼”

#### 9.7.2 需求改写

如果用户说的不是标准翡翠求购语句，例如偏场景表达、送礼表达、搭配表达，系统会先让 MiMo 改写成一个翡翠购物问题。

这样做的意义：

- 提高后续匹配稳定性
- 保证系统能接住“非行业表达”

#### 9.7.3 需求画像落库

解析后的需求会保存成 `visitor_need_profiles`，用于：

- 后续查看用户真实需求结构
- 让会话消息与匹配结果绑定

#### 9.7.4 商品匹配策略

优先级：

1. 先走向量检索
2. 如果 embedding 失败或无候选，再走规则匹配

规则匹配依据：

- 标题
- 简介
- 详情
- `search_text`
- 标签

评分因素：

- 向量距离
- 颜色/种水/器型/用途命中
- 关键词命中
- 价格接近度
- 商家是否 VIP
- 商品最近更新时间

#### 9.7.5 最终用户可见回复

系统不会把内部结构化信息直接吐给用户，而是再调一次 MiMo，生成一段自然语言回复。

如果 AI 失败，则使用模板兜底文案。

### 9.8 `vip_orders.py`

作用：

- 定义 VIP 套餐
- 计算金额
- 生成订单号
- 支付成功后给商家延长 VIP

续费逻辑：

- 如果商家当前 VIP 还没过期，则从当前到期日继续叠加
- 如果已过期，则从支付时间起算

### 9.9 `alipay.py`

作用：

- 支付宝参数签名
- 创建 PC/WAP 支付链接
- 查单
- 支付回调验签

它没有依赖第三方支付宝 SDK，而是自己做了：

- RSA2 签名
- 公钥验签
- 回调参数验签

这代表可控性高，但也要求密钥文件、验签规则和错误处理必须非常谨慎。

## 11. 买家端功能详解

### 11.1 AI 聊天找货首页

页面：`HomePage.tsx`

作用：

- 买家输入翡翠需求
- 系统实时回复
- 匹配商品卡片

前端实现：

1. 首次进入生成 `visitorId`
2. 本地缓存 `sessionId`
3. 本地缓存历史消息
4. 提交消息时先本地插入临时消息
5. 调 `/api/chat/sessions/{id}/matches/stream`
6. 按 SSE `message_delta` 逐步拼接回复
7. 收到 `match_result` 后把临时消息替换成真实消息
8. 如果旧 session 失效返回 404，则新建 session 重发

后端实现：

1. 创建或读取会话
2. 对输入做需求解析/改写
3. 生成结构化画像
4. 查匹配商品
5. 保存用户消息和助手消息
6. 以 SSE 形式分块回传

工作流程：

买家输入需求 → 前端发送流式请求 → 后端匹配商品 → 返回回复文本 + 商品卡片 → 前端展示聊天气泡和卡片。

### 11.2 商品卡片跳转

聊天卡片点击后，前端跳转到：

- `/products/:id`

这里是买家端从“聊”进入“看货”的承接。

### 11.3 商品详情页

页面：`ProductDetailPage.tsx`

作用：

- 展示商品大图、价格、标签、简介、详情
- 允许买家留邮箱联系卖家
- 支持分享

前端实现：

1. 调 `/api/products/{id}`
2. 读取商品详情
3. 支持图片轮播和大图预览
4. 提交邮箱到 `/api/products/{id}/contact`

后端实现：

1. 只允许读取 `status=listed` 的商品
2. 查询商家等级，返回 `merchantTier`
3. 买家留资后创建：

## 15. 已发现的问题与风险

下面按优先级从高到低列出。

### 15.1 高风险：生产敏感信息已进入仓库

证据：

- `deploy/.env.production`
- `deploy/secrets/alipay-app-private-key.pem`
- `deploy/secrets/alipay-public-key.pem`

问题性质：

- 生产数据库连接
- 应用密钥
- AI 平台 key
- 支付密钥文件

都不应该直接进代码仓库。

影响：

- 仓库泄露即等于生产密钥泄露
- 数据库、支付、AI 调用都可能被冒用

建议：

1. 立即轮换所有已暴露密钥
2. 把 `deploy/.env.production` 和 `deploy/secrets/*` 从仓库移除
3. 改为服务器环境变量或密钥管理机制

### 15.2 高风险：仓库部署配置与线上真实端口不一致

证据：

- `deploy/docker-compose.prod.yml` 使用 `6140/6141`
- `deploy/nginx.merchant.lyhlz.cn.conf` 代理到 `6140/6141`
- 用户确认线上后端实际为 `6145`

影响：

- 新人按仓库部署文档操作会直接配错
- 线上排查时容易出现“代码没问题但访问错端口”

建议：

1. 明确一份“线上事实配置”
2. 仓库内保留的部署文件要么更新成现状，要么注明“仅示例”

### 15.3 高风险：正式接口里混入自动 seed 演示数据

证据：

- `FZ-backend/app/api/routes/merchant.py` 中 `_seed_products_if_empty`
- 同文件 `_seed_leads_if_empty`
- 同文件 `_seed_notifications_if_empty`

影响：

- 新商家打开后台就可能自动生成假商品、假客资、假通知
- 演示逻辑和正式业务逻辑混在一起

建议：

1. 生产环境关闭 seed
2. 改成初始化脚本或演示模式开关

### 15.4 中风险：追加图片会清空当前商品草稿文案

证据：

- `FZ-backend/app/api/routes/merchant.py` 中 `append_product_draft_images`
- 追加图片时若草稿已存在，会把 `title/summary/detail/tags/price_cents` 直接清空

影响：

- 商家如果已经 AI 生成并手工编辑过内容，再补一张图，原文案会被清空
- 容易造成“我只是补图，为什么文案丢了”

这可能是设计选择，但从用户体验看更像数据丢失风险。

### 15.5 中风险：删除图片也会把商品文案整体清空

证据：

- `FZ-backend/app/api/routes/merchant.py` 中 `delete_product_image`

删除图片后会清空：

- `title`
- `summary`
- `detail`
- `tags`
- `price_cents`

影响：

- 用户删除一张图，整份商品草稿内容一起失效

如果这是为了逼用户重新生成，也应该通过明确状态提示表达，而不是静默清空。

### 15.6 低到中风险：商品状态更新中的 `published_at` 判断是无效分支

证据：

- `FZ-backend/app/api/routes/merchant.py`
- 条件：`if payload.status != "listed" and product.published_at is None: product.published_at = None`

这个条件在语义上基本没有效果：

- 只有本来就是 `None` 时才再赋值 `None`

说明：

- 这段逻辑要么是残留代码
- 要么作者原本想表达“下架时是否清理发布时间”，但最终没写完整

### 15.7 低风险：通知设置中的网页通知实际上不可关闭

证据：

- `MerchantProfileOut` 返回时固定 `web_notification_enabled=True`
- 更新通知设置时也强制 `merchant.web_notification_enabled = True`

影响：

- 数据库字段存在，但当前功能上没有真正的“网页通知开关”
- 前端看到的是固定开启

这更像“功能未完成”，不是致命 bug。

### 15.8 低风险：路由说明表和真实路由不一致

证据：

- `FZ-front/src/routes.ts` 写的是 `/product/:id`
- `FZ-front/src/App.tsx` 真实实现的是 `/products/:id`

影响：

- 占位页/文档路由说明不准确
- 后续若有人按说明表扩展功能会走错

### 15.9 低风险：代码风格检查当前不通过

本次执行结果：

- 前端 `npm run lint`：通过
- 前端 `npm run build`：通过
- 后端 `ruff check .`：未通过

失败点包括：

- `FZ-backend/alembic/versions/202606140001_vip_orders.py`
- `FZ-backend/app/api/routes/merchant.py`

这不是业务致命问题，但说明当前工作区代码质量检查并非全绿。

## 19. 建议的后续处理顺序

如果你下一步准备继续推进，我建议优先级如下：

1. 先处理敏感信息泄露和密钥轮换
2. 再统一线上/仓库部署配置
3. 把 seed 逻辑从正式接口中剥离
4. 修正“补图/删图导致文案被清空”的流程策略
5. 补齐后端测试环境，确保可在独立 test DB 跑通
6. 最后再做细节体验优化和功能扩展
