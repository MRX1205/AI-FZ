# 高翠网三大核心业务排查手册

更新时间：2026-06-16

这份文档给开发、测试、运维排查问题时使用。

目标：

1. 明确三条主业务链每一步该看哪里
2. 出问题时先查哪个接口、哪个表、哪个代码文件
3. 快速判断问题是在前端、后端、AI 服务、支付回调还是数据库

---

## 1. 快速总览

三条核心业务链：

1. 买家找货
2. 商家发布商品
3. VIP 支付

核心判断原则：

- 页面没反应：先看前端请求是否发出
- 请求报错：先看后端接口逻辑和返回 detail
- 接口成功但结果不对：再看服务层和数据库落库
- 线上偶发异常：优先排环境变量、文件路径、第三方服务连通性

---

## 2. 买家找货排查

## 2.1 标准链路

1. 首页创建或恢复 chat session
2. 前端请求 `/api/chat/sessions/{id}/matches/stream`
3. 后端调用 `match_products_for_session(...)`
4. 必要时做需求改写
5. 解析需求画像
6. 落库 `visitor_need_profiles`
7. 向量检索或规则检索商品
8. 生成可见回复
9. 落库 `chat_messages`
10. SSE 回前端
11. 前端展示商品卡片
12. 点商品详情后，提交邮箱留资

---

## 2.2 先看哪些前端文件

- 首页聊天页：
  `FZ-front/src/pages/HomePage.tsx`
- 商品详情页：
  `FZ-front/src/pages/ProductDetailPage.tsx`

重点看：

1. 是否成功创建 `sessionId`
2. 是否请求了 `/matches/stream`
3. 浏览器网络面板里 SSE 是否持续回包
4. `match_result` 事件是否有数据
5. 商品卡片点击后是否跳详情页
6. 详情页提交邮箱后是否请求 `/api/products/{id}/contact`

---

## 2.3 先看哪些后端文件

- 聊天接口：
  `FZ-backend/app/api/routes/chat.py`
- 商品详情和留资接口：
  `FZ-backend/app/api/routes/products.py`
- 找货核心服务：
  `FZ-backend/app/services/visitor_product_matcher.py`
- MiMo 封装：
  `FZ-backend/app/services/jade_agent.py`
- 向量服务：
  `FZ-backend/app/services/embeddings.py`

## 2.4 数据库要看哪些表

- `visitor_need_profiles`

## 2.5 常见问题与排查顺序

### 问题 A：首页发送后一直转圈，没回复

先查：

1. 浏览器里 `/api/chat/sessions/{id}/matches/stream` 是否返回 200
2. 如果不是 200，看 response body 的错误 detail
3. 如果是 200，但没有 SSE 数据：
   - 查后端是否卡在 MiMo
   - 查后端是否卡在 embedding

重点文件：

- `chat.py`
- `visitor_product_matcher.py`
- `jade_agent.py`
- `embeddings.py`

可能原因：

1. MiMo key 配置错误
2. DashScope key 配置错误
3. 商品 embedding 表为空，导致只能规则匹配
4. 流式响应被 Nginx 缓冲或中断

### 问题 B：回复出来了，但没有商品卡片

先查：

1. `chat_messages.matched_products` 是否为空
2. `visitor_need_profiles` 是否成功落库
3. `merchant_products` 是否存在 `status = listed` 的真实商品
4. `merchant_product_embeddings` 是否有对应商品向量

可能原因：

1. 没有已上架商品
2. 商品没有生成 embedding
3. 需求解析过弱，关键词没命中
4. 向量匹配失败后规则匹配仍无候选

### 问题 C：商品详情能打开，但留资失败

先查：

1. `/api/products/{id}/contact` 的返回码
2. `merchant_leads` 是否生成记录
3. `merchant_notifications` 是否同步生成
4. 同邮箱是否已提交过同商品

可能原因：

1. 商品不是 `listed`
2. 邮箱格式前端校验未通过
3. 数据库提交失败
4. 已有重复 lead，被查重拦截

---

## 2.6 买家找货链路最重要的 5 个检查点

1. `chat_sessions` 是否创建成功
2. `visitor_need_profiles` 是否落库
3. `merchant_products` 是否有已上架真实商品
4. `merchant_product_embeddings` 是否存在
5. `chat_messages.matched_products` 是否有值

只要这 5 个点串起来，链路通常就是通的。

---

## 3. 商家发布商品排查

## 3.1 标准链路

1. 商家进入发布页
2. 请求 `/api/merchant/products/current-draft`
3. 上传图片到 `/api/merchant/products/drafts/images`
4. 图片写入磁盘和 `merchant_product_images`
5. 点击 AI 生成，调用 `/api/merchant/products/drafts/generate`
6. 后端读取图片并调用 MiMo 多模态
7. 生成商品草稿并写入 `merchant_products`
8. 商家修改后点击发布
9. 调 `/api/merchant/products/{id}/publish`
10. 校验完整性、额度、embedding
11. 商品变成 `listed`
12. 写入或更新 `merchant_product_embeddings`

---

## 3.2 先看哪些前端文件

- 发布页：
  `FZ-front/src/pages/MerchantPublishPage.tsx`
- 编辑页：
  `FZ-front/src/pages/MerchantProductEditPage.tsx`
- AI 结果页：
  `FZ-front/src/pages/MerchantPublishResultPage.tsx`

先看：

1. `current-draft` 是否返回草稿
2. 图片上传接口是否成功
3. AI 生成接口是否返回完整商品对象
4. 发布接口是否成功

---

## 3.3 先看哪些后端文件

- 商家接口：

## 3.4 数据库与文件系统要看哪些

数据库表：

- `merchant_products`
- `merchant_product_images`

4. `merchant_product_images.public_url`
5. `merchant_product_embeddings.product_id`

## 3.5 常见问题与排查顺序

### 问题 A：上传图片成功，但 AI 生成报 400/500

先查：

1. `/api/merchant/products/drafts/generate` 返回 detail
2. 图片是否真的是 `/uploads/...`
3. 上传后的物理文件是否存在
4. MiMo API key 是否可用

重点看：

- `product_image_recognition_agent.py`

常见原因：

1. 图片没真正写入服务器目录
2. Nginx 公网可访问路径和后端本地实际路径不一致
3. MiMo 返回非 JSON
4. MiMo 返回字段不完整

### 问题 B：AI 生成成功，但发布失败

先查：

1. 商品是否有图片
2. `title/summary/detail/price_cents` 是否完整
3. 免费商家是否超过 2 件额度
4. embedding 是否生成失败

重点看：

- `merchant.py` 中的 `_assert_product_ready_to_publish`
- `_assert_product_quota_available`
- `_refresh_product_embedding`

常见原因：

1. 商品价格为 0
2. 标题简介详情为空
3. 发布额度超限
4. DashScope embedding 服务失败

### 问题 C：发布成功了，但买家端匹配不到

先查：

1. 商品是否 `status = listed`
2. `merchant_product_embeddings` 是否存在记录
3. `search_text` 是否生成为空或极弱
4. 商品标签和 `match_params` 是否明显异常

常见原因：

1. 商品其实还是 `draft`
2. embedding 没写成功
3. AI 草稿内容质量太差
4. 发布后又被下架

---

## 3.6 商家发布链路最重要的 6 个检查点

1. 商品图片物理文件是否存在
2. `merchant_product_images` 是否有记录
3. `merchant_products.image_urls` 是否已写入
4. `merchant_products.search_text` 是否存在
5. `merchant_products.status` 是否为 `listed`
6. `merchant_product_embeddings` 是否存在

## 4. VIP 支付排查

## 4.1 标准链路

1. 商家打开账户权限页
2. 请求 `/api/merchant/account`
3. 选择套餐
4. 请求 `/api/merchant/account/vip-orders`
5. 后端创建订单并返回 `payUrl`
6. 浏览器跳支付宝
7. 支付宝异步回调 `/api/payments/alipay/notify`
8. 后端更新 `merchant_vip_orders`
9. 更新商家 VIP 状态
10. 前端回到支付结果页
11. 前端调用 `/vip-orders/{id}` 和 `/vip-orders/{id}/sync`
12. 若回调延迟，主动查单补同步

---

## 4.2 先看哪些前端文件

- 账户权限页：
  `FZ-front/src/pages/MerchantAccountPage.tsx`
- 支付结果页：
  `FZ-front/src/pages/MerchantAccountPaymentResultPage.tsx`

先看：

1. `/api/merchant/account` 是否正常返回
2. `/api/merchant/account/vip-orders` 是否成功返回 `payUrl`
3. 支付结果页是否拿到了 `order_id`
4. `/sync` 是否能正常返回订单状态

---

## 4.3 先看哪些后端文件

- 商家支付接口：
  `FZ-backend/app/api/routes/merchant.py`
- 支付宝回调接口：
  `FZ-backend/app/api/routes/payments.py`
- VIP 订单服务：
  `FZ-backend/app/services/vip_orders.py`
- 支付宝客户端封装：
  `FZ-backend/app/services/alipay.py`

---

## 4.5 常见问题与排查顺序

### 问题 A：点击升级 VIP 没有跳支付宝

先查：

1. `/api/merchant/account/vip-orders` 是否报错
2. 返回体中是否有 `payUrl`
3. 支付宝配置项是否完整

常见原因：

1. `ALIPAY_APP_ID` 未配置
2. 私钥/公钥文件路径错误
3. `BACKEND_PUBLIC_BASE_URL` 或 `FRONTEND_PUBLIC_BASE_URL` 配置不对
4. 生成 page/wap pay url 失败

### 问题 B：支付成功了，但商家还是普通会员

先查：

1. `/api/payments/alipay/notify` 是否收到回调
2. 回调是否验签成功
3. 订单金额是否一致
4. `merchant_vip_orders.status` 是否已变 `paid`
5. `merchants.tier` 是否已改成 `vip`

如果回调没到，再查：

1. 支付结果页的 `/sync` 是否成功
2. `query_trade` 是否能查到 `TRADE_SUCCESS`

常见原因：

1. 回调地址公网不可达
2. Nginx 没正确转发 `/api/payments/alipay/notify`
3. 验签失败
4. 支付金额校验失败
5. 订单号不匹配

### 问题 C：续费后有效期不对

先查：

1. 支付时商家是否仍是有效 VIP
2. `grant_started_at` 是否从旧过期时间顺延
3. `grant_expires_at` 是否正确加月

重点看：

- `vip_orders.py` 的 `mark_vip_order_paid`
- `add_months`

---

## 4.6 VIP 支付链路最重要的 5 个检查点

1. `merchant_vip_orders` 是否创建成功
2. 支付链接 `payUrl` 是否返回
3. 支付宝回调是否到达
4. `merchant_vip_orders.status` 是否变成 `paid`
5. `merchants.tier/vip_expires_at` 是否更新

---

## 5. 三条业务联调时的总排查顺序

如果线上反馈一句“功能不能用”，建议按这个顺序排：

1. 看浏览器网络请求，先确认是哪个接口失败
2. 看接口返回 detail，不先猜
3. 看对应后端路由文件
4. 看对应服务层文件
5. 看数据库是否真正落库
6. 涉及 AI 时查 MiMo/DashScope 配置和连通性
7. 涉及上传时查文件物理路径和 Nginx 映射
8. 涉及支付时查回调是否真正打到后端

---

## 6. 线上最值得优先监控的点

如果后续要做更稳的生产监控，最建议优先加这几类日志和告警：

1. `/api/chat/sessions/*/matches/stream` 的错误日志
2. MiMo 接口失败日志
3. DashScope embedding 失败日志
4. `/api/merchant/products/drafts/generate` 的失败日志
5. `/api/payments/alipay/notify` 的验签失败日志
6. `merchant_vip_orders` 长时间 pending 的订单日志

---

## 7. 一句话总结

开发排查时，不要把问题笼统看成“AI 不行”或“支付不行”，而要拆成一条业务链中的具体节点：请求有没有发出、后端有没有接住、服务有没有跑完、数据库有没有落、状态有没有回到前端。
