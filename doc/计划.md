# 高翠网开发计划

版本：V1.1
技术方向：移动端 H5 + Python FastAPI + PostgreSQL/pgvector

## 一、目标与成功标准

本项目先按已有 PRD 和原型图开发一个移动端 H5 产品闭环：买家通过 AI 聊天找翡翠，查看商品详情并提交意向；商家登录后管理商品、发布商品、查看客资、管理账户权限。

成功标准：

1. 已有原型图覆盖的页面逐页开发，每页完成后单独验收视觉、交互、跳转。
2. 缺失原型图的页面只做路由占位，不擅自补完整 UI，等补图后再开发。
3. 前端页面先能使用 mock 数据跑通闭环，后端接口完成后切换真实 API。
4. 后端具备商家、商品、图片、标签、客资、聊天、通知、会员权限等核心数据能力。
5. AI 接入预留清晰接口，后续支持商品图片识别、商品文案生成、买家需求解析、商品匹配。

## 二、总体技术栈

### 前端

- React + Vite + TypeScript
- React Router
- lucide-react 图标
- CSS Modules 或普通 CSS，先不引入大型 UI 组件库
- 移动端 H5 适配，重点视口：390x844、430x932

### 后端

- Python 3.12+
- FastAPI
- SQLAlchemy 2.x
- Alembic
- Pydantic
- Uvicorn/Gunicorn
- Redis
- Celery 或 RQ，用于 AI 生成、图片识别、embedding 生成等异步任务

### 数据库与存储

- PostgreSQL 16+
- pgvector：商品语义向量检索
- Redis：验证码、限流、短期会话、任务队列
- S3 兼容对象存储：商品图片、AI 识别用原图
- 本地开发可使用 MinIO 替代对象存储

数据库选择理由：

1. PostgreSQL 适合商家、商品、客资、会员权限这类关系型业务。
2. pgvector 可以直接在 PostgreSQL 内做商品语义匹配，首版不需要单独部署向量数据库。
3. 后续商品量或向量检索压力上来后，再考虑拆出 Qdrant，不在首版过度设计。

```text

/product/:id                   商品详情
/merchant/auth                 商家登录/注册
/merchant/dashboard            商家后台

/merchant/publish/result       发布商品-AI生成结果

/merchant/products/:id/edit    商品编辑
```

前端公共模块：

- `store/`：登录状态、商家身份、聊天记录、发布商品草稿

## 五、后端模块计划

### 1. 认证与商家模块

能力：

- 邮箱验证码登录/注册
- 商家登录态
- 商家会员类型：免费、VIP
- VIP 到期时间
- 退出登录

核心接口：

```text
POST /api/auth/send-code
POST /api/auth/login
POST /api/auth/logout
GET  /api/auth/me
GET  /api/merchant/profile
PATCH /api/merchant/profile/email
PATCH /api/merchant/notification-settings
```

### 2. 商品模块

能力：

- 商品图片上传
- AI 生成商品标题、简介、详情、标签、预估价
- 商品发布
- 商品列表筛选：全部、已上架、草稿、已下架
- 商品编辑
- 商品上下架
- 商品删除
- 免费商家最多 2 件，VIP 商家最多 100 件

核心接口：

```text
POST   /api/uploads/product-images
POST   /api/products/ai-generate
POST   /api/products
GET    /api/products
GET    /api/products/{product_id}
PATCH  /api/products/{product_id}
PATCH  /api/products/{product_id}/status
DELETE /api/products/{product_id}
```

### 3. 买家 AI 匹配模块

能力：

- 首页多轮聊天
- 判断是否属于翡翠需求
- 解析预算、品类、尺寸、品相、用途等信息
- 匹配 3 个商品卡片
- 非翡翠需求返回固定提示文案
- 聊天记录持久化

核心接口：

```text
POST /api/chat/sessions
GET  /api/chat/sessions/{session_id}/messages
POST /api/chat/sessions/{session_id}/messages
```

匹配流程：

1. 接收用户自然语言需求。
2. 判断是否属于翡翠服务范围。
3. 使用 LLM 或规则解析结构化需求。
4. PostgreSQL 先按状态、价格、标签、商家权限做硬筛。
5. pgvector 按商品标题、简介、详情、标签 embedding 做语义召回。
6. 按 VIP 权重、价格接近度、标签命中数、更新时间重排。
7. 返回最多 3 个商品卡片。

### 4. 客资模块

能力：

- 买家在商品详情页提交邮箱和留言
- 生成客资并归属到对应商家
- 商家查看客资列表
- 免费商家邮箱打码
- VIP 商家查看完整邮箱
- 标记已联系
- 复制邮箱由前端调用剪贴板，后端只提供完整邮箱权限校验

核心接口：

```text
POST  /api/leads
GET   /api/merchant/leads
GET   /api/merchant/leads/{lead_id}
PATCH /api/merchant/leads/{lead_id}/status
```

### 5. 账户权限模块

能力：

- 查询当前会员权限
- 查询发布额度
- 查询升级/续费套餐
- 联系运营升级/续费先记录意向，弹窗原型补齐后再做完整交互

核心接口：

```text
GET  /api/merchant/account/permissions
GET  /api/merchant/account/plans
POST /api/merchant/account/upgrade-intent
```

### 6. 通知模块

能力：

- 新客资通知
- VIP 到期前 30 天提醒
- 系统通知列表

核心接口：

```text
GET   /api/merchant/notifications
PATCH /api/merchant/notifications/{notification_id}/read
```

## 七、AI 接入计划

### 首版 mock 阶段

前端和后端先使用固定返回：

- 首页翡翠需求：返回解析文本 + 3 个商品卡片。
- 首页非翡翠需求：返回 PRD 指定固定文案。
- 商品 AI 生成：根据上传图片返回固定标题、简介、详情、标签、价格。

### API 接入阶段

AI 能力拆成 4 个函数，后续替换供应商时只改服务层：

```text
classify_jade_intent(text)
parse_buyer_need(text)
generate_product_copy(image_urls)
embed_product(product)
```

### 商品匹配策略

首版不做复杂推荐系统，只做可解释的混合匹配：

1. 硬条件过滤：商品已上架、价格范围、标签、商家可展示。
2. 语义召回：pgvector 查找与用户需求最相近的商品。
3. 业务排序：VIP 优先、标签命中更多优先、价格更接近优先、更新时间更近优先。
4. 返回 3 个商品。

## 九、PRD 冲突处理

1. 商品详情原型只有邮箱输入，PRD 要邮箱和留言两个必填框：按 PRD 做两个输入框。
2. 商品编辑原型顶部为“删除/保存”，PRD 要底部“上/下架商品 + 保存修改”：按 PRD 做底部按钮。
3. PRD 要“每页所有内容一屏内展示完”，但长内容页面不可读：采用关键首屏策略，长内容允许滚动，底部主操作固定。
4. 系统通知页 PRD 指向 13，但现有 13 是个人中心：通知页先占位。
5. PRD 提到前后端方案、AI-PIPELINE、买家匹配 AGENT 设计，但当前项目目录没有这些文件：先按本文方案开发，后续若补充外部方案，再对比冲突并调整。

## 十、默认假设

1. 商家登录首版只做邮箱验证码登录，不做密码。
2. 默认测试账号：`seller@email.com` 为 VIP，`free@email.com` 为免费商家。
3. 金额后端用分存储，前端显示为人民币格式。
4. 免费商家最多发布 2 件商品，VIP 商家最多发布 100 件商品。
5. 免费商家看不到完整买家邮箱，VIP 可见完整邮箱。
6. 商品匹配只返回已上架商品。
7. 缺少原型图的页面不做完整设计，只保留可跳转占位页。
