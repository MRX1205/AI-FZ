## 1. 核心技术栈与基础设施

### 1.1 基础设施架构
*   **PostgreSQL + pgvector**: 核心关系型数据库。利用 `pgvector` 扩展存储大模型生成的 `[1024]` 维浮点向量，支撑语义级别（而非单纯的全文检索）商品精准召回。
*   **Redis**: 声明于 `docker-compose.yml` 中，架构上为防刷限流、高频缓存、异步任务队列预留身位。
*   **MinIO / 文件系统**: 现阶段使用 `uploads` 文件夹直接挂载 FastAPI 静态路由 `/uploads`，用于实际存储商家的海量商品图片等。
*   **大语言与多模态视觉模型**: 
    1.  **MiMo/DashScope (文本)**: 驱动 `Jade Agent` 与游客聊天并解析意图。
    2.  **DashScope 视觉模型**: 驱动后台商户识图“一键自动填充参数上架并打标签”。
    3.  **DashScope Embeddings**: 对长段商品预料执行特征抽取，生成 1024 维向量。

### 1.2 后端应用架构 (FZ-backend)
*   **FastAPI & Uvicorn**: 异步高并发 HTTP 服务入口。`app/main.py` 通过 `CORSMiddleware` 处理跨域。
*   **SQLAlchemy 2.0 (Async) + Alembic**:
    *   重度使用 SQLAlchemy 2.0 的 `Mapped` 和 `mapped_column` 提供强类型映射诊断。
    *   `alembic/versions` 管理了多达十几套迁移（如 `202606120009_product_embedding_hnsw.py` 专门为向量检索加持了高阶 HNSW 索引）。
*   **依赖强校验**: 采用 `pydantic-settings` 统一读取 `.env.bak.codex` (核心机密配置如 `SUPABASE_ANON_KEY`, `ALIPAY_APP_ID`, `MIMO_API_KEY`)。

### 1.3 前端应用架构 (FZ-front)
*   **React 19 + TypeScript + Vite**: 采取最高配和前沿的构建组合。约束于 430px 内模拟移动端（H5）排版。
*   **React Router 7 分流策略**: 在主路由文件 `src/routes.ts` 中维护了 15 个独立页面入口方案，用 `implementedRoutes` 集合机制降级拦截未完成页至 `PlaceholderPage`。
*   **去状态化架构**: 没有运用 Redux / Zustand 这类集中库。完全使用原生的 React Context, 配合自定义封装的 `fetch` API (`src/api/client.ts`) 以及 `localStorage` (`merchantAuthStorage.ts`) 控制状态，极大减小了应用打包体积。

## 2. 数据库字典及深度字段关联 (Data Models)

系统的数据表建立在 PostgreSQL 基础上，其外键和索引关系十分明确：

### 2.1 商家与提权模块 (`merchant.py` / `payment.py`)
*   **`merchants` 表**: 除邮箱等常规信息，独含 `tier (枚举: free / vip)`。当 `tier` 被置为只读并判定时，整个后台展示流均以此作为拦截依据。
*   **`vip_orders`** *(对应文件202606140001_vip_orders.py)*: 订单流水表。关联支付宝交易结果，成功则更新对应 merchant 的 `vip_started_at` 并重置或延续到期时间。

### 2.2 大模型语义与交互承接模块 (`chat.py`)
*   **`chat_sessions` 表**: 作为游客指纹落地锚点，拥有 `visitor_id` (游客用 uuid)。所有的对话上下文追溯都基于这个会话 ID 捞取历史(`history`)传入大模型。
*   **`chat_messages` 表**: 外键链指 Session。其带有强大的 `matched_products (JSONB)` 字段——这意味着并非只抛给用户“干巴巴的话”，一旦数据库计算出了对应的翡翠向量近邻结果，就会存为推荐 JSON 附生在该消息之下发往前端渲染卡片。
*   **`visitor_need_profiles` 表**: 定时把海量无意义的聊天归纳为真正的“找货雷达模型”。内含 `normalized_question`（被格式化的请求语）以及 `params`（AI 帮你提取出的需求结构，如：`{种水:冰种, 颜色:紫罗兰}`）。这也是发起 HNSW 算法匹配的触发引线。

### 2.3 向量级商品档案模块 (`product.py`)
*   **`merchant_products` 表**: 存储 `price_cents` (强制以分为单位规避精度浮点流失)，具备 `match_params` 字典列。
*   **`merchant_product_embeddings` 表**:
    *   **核心特性列**: `embedding: Vector(1024)`。当调用 Dashscope 的 Embeddings API 收到浮点矩阵推入当前列。
    *   **快速召回索引机制**: 基于 `HNSW` (Hierarchical Navigable Small World) 构建距离检索网，它替代了传统关系型数据库 LIKE 匹配失效的问题，支持使用余弦相似距离 `<=>` 操作符等进行极速比对。

### 2.4 买家留资转化模块 (`lead.py`)
*   **`merchant_leads` 表**: 专门捕获买手填写的表单。包含了敏感数据列 `buyer_email`。在 Controller 层，如果读出当前关联商家的 `MerchantTier.free`，代码会在吐给前端时强制抹去明文并替换成 `***@***.com` 以诱导支付扩容。

## 3. 全局业务架构：三大核心智能 Agent 及 Service

该项目在常规 MVC 以外挂载了一层极为硬核的 AI 服务代理 (`app/services`)：

### 3.1 买家对客代理 (Jade Agent - `jade_agent.py`)
*   **机制**: `httpx.AsyncClient` 高速发送 HTTP POST 携带 API Key 请求给设定好的模型终点。
*   **控制与纠偏**: AI 最怕“随口乱答应造假大空数据”。通过硬编码的 `MIMO_SYSTEM_PROMPT` 下达铁令：“*你是专业助手...不要承诺已经找到具体商品，也不要输出商品卡片...不要自己编造数据*”。若有任何连接错误如超时、欠费，直接返回定制好的兜底文案 (`MIMO_ERROR_REPLY`)，确保对客不裸报错。

### 3.2 向量近邻推荐代理 (`embeddings.py` 与 `visitor_product_matcher.py`)
*   **运行工作流**: 
    1.  当发生新的意图生成或存量商品重置编辑时调用 Dashscope `/embeddings` 将文字打成浮点。
    2.  `visitor_product_matcher.py` 去查询那些和 `visitor_need_profiles` 取出来的参数匹配的库中库存。
    3.  利用 SQLAlchemy 的原生操作直接在底层发起欧氏/内积数据库级别的计算查询并分页 Limit 出最近商品返向前端推送层。

### 3.3 商户省心识图代理 (`product_image_recognition_agent.py`)
*   **机制**: 在前端执行 H5 本地抓取摄像头原石/戒面并传入 `UPLOAD_BASE` 中。
*   **具身视觉触发**: Python 执行 File 读取并 `base64` 编码塞接通义系（或其他）大视觉识别网络。
*   **逆推解析 Prompt**: 携带指令如 `PRODUCT_IMAGE_RECOGNITION_AGENT_ROLE` （不凭空编造证书、看不出来的尺寸写“未知”，不写夸大文案），其结果通过强解析抽出，最后让商家到达 `MerchantPublishResultPage` 以免误判。

## 4. 其余关键支撑体系深度拆解

### 4.1 防撞库无密认证 (Supabase OTP)
由于对接自建的基于发信、时效与流控管理的邮件或短信太重，因此通过调用 Supabase 封装好底层的 `OTP (/auth/v1/otp)` 和 `Verify (/auth/v1/verify)`。当检测到 400（验证码伪造错误）、429（恶意刷攻击）都会被 Python 类 `SupabaseOtpClient` 手动 catch 并转化出友好的中文给前端抛 `HTTPStatusError` 提醒。

### 4.2 支付宝（Alipay）RSA 不对称支付体系
*   在 `app/services/alipay.py` 构建。未使用老旧及庞大的 SDK 依赖包，人工对接。
*   **发包**: 前端触发 VIP 选项，向后生成全站唯一串作为 `out_trade_no`（流水），通过私钥加密发给阿里的接口并携带跳转指令。
*   **可靠异步监听**: 暴露 `Notify URL` 钩子供外网支付宝请求成功事件 `trade_status == TRADE_SUCCESS`，验证对方支付宝传过来的公钥无虚假篡改（`AlipayVerificationError` 拦截），放行该请求并将原库中的提权字段写入存量商户身上，前台即可秒刷去除 Leads 买家邮箱的星号遮挡。

## 5. 项目工程评估及风险建议

**该架构虽然小巧，但极度精准地卡住了“商品多模态入库”+“消费者口语意图解析出库”这两个核心痛点，并利用 pgvector 搭建了企业级的商品召回管道。**

作为接手的专业系统工程师，特在此指出隐患需重点观测的地方：

1.  **AI调用财务流失陷阱 (Rate Limit Missing)**: 对话接口(`jade_agent`) 与识图上架高频依赖三方服务(MiMo/DashScope)，代码目前没有检测到如 Redis 计数的令牌桶 `Limiter`。需要注意提防不法恶意者写脚本无休止刷 `Chat/Completions` 耗干 API 资金余额。
2.  **遗留碎片文件地雷 (Orphaned Files)**: 商家点开发布图但断网了，或上传却不确定的商品图片，将永远堆在服务器本地 `/uploads/...` UUID 文件夹下。必须后期接入 Celery 定时任务或 Cron 脚本当 `status=draft` 时间过长的垃圾图片定时擦除，以防本地磁盘挤爆。
3.  **HNSW 索引优化**: 1024 维特征检索如果发生在大十几万商品的存量店铺里，即便有索引，依然会吃巨大的 PG 运行时刻 Shared Memory。要防范 OOM 宕机（Out of memory 现象）。

（本版本在架构层面、表字段意义和 API 推衍层面均执行了深度详单拆解复盘，适合硬核研发阅读。）
